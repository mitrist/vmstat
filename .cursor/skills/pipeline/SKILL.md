# Pipeline: Диспетчер → Разработчик → QA

Реализация концепции из [agents.md](agents.md). Роли **Разработчик** и **QA** исполняет агент Cursor по этому сценарию; состояние плана можно вести вручную или через [agents/dispatcher.py](agents/dispatcher.py).

## Когда включать

Пользователь пишет `@pipeline` с планом, например:

`@pipeline план: 1. Добавить endpoint /health 2. Написать тест`

Или даёт план в следующем сообщении после `@pipeline`.

## Роли (три агента)

### 1. Диспетчер

- Держит **план** (упорядоченный список строк-задач) и **индекс** текущего пункта (с 0).
- `getNextTask()` — вернуть текущий пункт или `ALL_DONE`.
- После вердикта QA: `receiveQAVerdict("OK")` — сдвинуть индекс; `"FAIL"` — индекс **не** менять.

### 2. Разработчик

- Получает **одну** строку задачи от диспетчера.
- Пишет/меняет код в репозитории, соблюдая стиль проекта.
- В конце явно: **«Готово к проверке: …»** (что сделано, какие файлы).

### 3. QA

- Сверяет результат с формулировкой задачи, просматривает синтаксис и явные регрессии.
- Итог: **`OK`** или **`FAIL`** + **конкретные** шаги исправления при FAIL.
- При `OK` сообщает, что диспетчер должен принять `receiveQAVerdict("OK")`. При `FAIL` — то же для `"FAIL"` (повтор той же задачи).

## Цикл скилла (обязательный порядок)

Пока диспетчер не вернул `ALL_DONE`:

1. **Диспетчер** — объявить текущий пункт плана (номер и текст).
2. **Разработчик** — выполнить только этот пункт; вывести «Готово к проверке».
3. **QA** — проверка; вывести вердикт **OK** или **FAIL** и пояснение.
4. Если **OK** — зафиксировать переход к следующему пункту (как после `receiveQAVerdict("OK")`).
5. Если **FAIL** — **не** переходить к следующему пункту; разработчик исправляет по замечаниям QA, затем снова шаг 3.

После `ALL_DONE` — краткий **итог**: какие файлы созданы/изменены, что осталось в бэклоге (если есть).

## Интерактивность

После каждого шага кратко показывать:

- что выдал диспетчер;
- что сделал разработчик (файлы, суть);
- вердикт QA и при FAIL — нумерованный список правок.

## Ссылка на код диспетчера

Для скриптов и тестов: класс `DispatcherAgent` в [agents/dispatcher.py](agents/dispatcher.py), метаданные ролей в [agents/definitions.json](agents/definitions.json).

## Пример мини-плана (из agents.md)

План: `["Написать функцию add(a,b)", "Написать тест для add"]` — первый пункт до OK QA, затем второй; при FAIL на тестах — доработка assert без смены пункта.
